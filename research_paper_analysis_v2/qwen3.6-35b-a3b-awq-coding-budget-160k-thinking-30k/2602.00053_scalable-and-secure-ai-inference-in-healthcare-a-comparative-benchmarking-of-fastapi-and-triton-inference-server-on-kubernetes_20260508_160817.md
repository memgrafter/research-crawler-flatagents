# Scalable and Secure AI Inference in Healthcare: A Comparative Benchmarking of FastAPI and Triton Inference Server on Kubernetes

## Quick Facts
- arXiv ID: 2602.00053
- Source URL: https://arxiv.org/abs/2602.00053
- Reference count: 17
- Primary result: Triton’s dynamic batching on GPU increases throughput by 73% over FastAPI with marginal latency overhead, validating a hybrid FastAPI-Triton architecture as optimal for regulated clinical AI.

## Executive Summary
This study benchmarks FastAPI and NVIDIA Triton Inference Server for deploying a DistilBERT sentiment analysis model within a Kubernetes-native healthcare architecture. Under controlled load testing, FastAPI running on CPU with a fixed batch size of 1 delivered the lowest p50 latency (22 ms) at 450 req/s, while Triton on GPU without batching achieved 28 ms p50 latency at 420 req/s. Enabling Triton’s dynamic batching (size 16) scaled throughput to 780 req/s (+73% over FastAPI) with a modest p50 increase to 34 ms and p95 latency of 60 ms.

The authors conclude that FastAPI minimizes overhead for single-request workloads, whereas Triton maximizes scalability through GPU-accelerated batching. Consequently, a hybrid architecture is validated as optimal: FastAPI functions as a secure PHI de-identification and routing gateway, while Triton handles high-volume backend compute. This pattern balances low-latency compliance handling with scalable inference performance for regulated clinical systems.

## Method Summary
Reproduction notes were not provided in the source material. The methodology is synthesized from the benchmarking parameters described in the study: a DistilBERT sentiment analysis model was containerized and deployed on Kubernetes, comparing FastAPI (CPU-bound, batch size 1) against NVIDIA Triton Inference Server (GPU-accelerated, with and without dynamic batching). Load was applied under controlled conditions to isolate the effects of hardware acceleration and batching strategies on inference latency (p50, p95) and request throughput (req/s). The evaluation also incorporated an architectural design for separating PHI processing from model inference to assess regulatory alignment.

## Key Results
- FastAPI on CPU (batch 1) achieves the lowest single-request latency (p50: 22 ms) but caps at 450 req/s.
- Triton with dynamic batching (size 16) scales throughput to 780 req/s (+73% over FastAPI) with p50: 34 ms and p95: 60 ms.
- A hybrid gateway-compute pattern (FastAPI for PHI handling, Triton for inference) optimally balances compliance isolation and GPU scalability.

## Why This Works (Mechanism)
No explicit mechanistic or onboarding analysis was provided in the source. Based on the benchmarking outcomes, the performance divergence is driven by how each framework handles request aggregation and hardware utilization. FastAPI’s lightweight Python web server adds minimal serialization and routing overhead, making it optimal for low-latency, single-request traffic. Triton, however, employs an optimized inference engine that aggregates incoming requests into dynamic batches, amortizing GPU kernel launch and memory transfer costs across multiple samples. The hybrid design leverages this by offloading security-sensitive PHI parsing to FastAPI while reserving Triton for compute-heavy batching, preventing compliance logic from bottlenecking GPU throughput.

## Foundational Learning
- **Dynamic Batching vs Fixed Batching**
  - *Why needed:* Fixed batches underutilize GPU compute when request volume is uneven; dynamic batching adapts to real-time load.
  - *Quick check:* Plot throughput vs latency across batch sizes 1, 8, 16, 32 to locate the performance knee point.
- **PHI Boundary Isolation**
  - *Why needed:* Healthcare regulations require strict separation of data handling and model execution to prevent data leakage.
  - *Quick check:* Verify network egress rules and container isolation to ensure raw PHI never crosses into the Triton namespace.
- **Kubernetes Resource Quotas in AI Inference**
  - *Why needed:* Unbounded GPU/CPU allocations cause node contention and latency tail-outs under bursty clinical traffic.
  - *Quick check:* Monitor p95 latency while gradually increasing pod replicas to validate quota enforcement.
- **Framework Serialization Overhead**
  - *Why needed:* Python-to-TensorRT conversion and gRPC/HTTP framing can introduce measurable latency if not optimized.
  - *Quick check:* Profile end-to-end request serialization costs across FastAPI endpoints vs Triton gRPC interfaces.

## Architecture Onboarding
- **Component Map:** Client Request -> FastAPI Gateway (PHI De-identification/Validation) -> Kubernetes Service Mesh -> Triton Inference Server (GPU Compute) -> Response Aggregation -> Client
- **Critical Path:** Inference latency is dominated by Triton GPU execution time and the dynamic batch window; FastAPI adds minimal routing overhead.
- **Design Tradeoffs:** FastAPI prioritizes lowest single-request latency and compliance isolation; Triton prioritizes high throughput via GPU batching at the cost of slightly higher p50/p95 latency.
- **Failure Signatures:** GPU OOM errors under sudden batch spikes, FastAPI thread exhaustion on traffic bursts, or Triton batching delays causing p95 latency tail-outs.
- **First Experiments:**
  1. Scale batch size from 1 to 32 to identify throughput vs latency breakpoints.
  2. Inject synthetic PHI payloads to measure FastAPI gateway de-identification overhead.
  3. Simulate GPU node failure and verify automatic request failover/re-routing to CPU-backed FastAPI.
