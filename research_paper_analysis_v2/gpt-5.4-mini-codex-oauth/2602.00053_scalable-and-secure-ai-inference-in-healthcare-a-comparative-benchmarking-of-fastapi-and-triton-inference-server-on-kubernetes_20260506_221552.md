# Scalable and Secure AI Inference in Healthcare: A Comparative Benchmarking of FastAPI and Triton Inference Server on Kubernetes

## Quick Facts
- **arXiv ID:** 2602.00053
- **Source URL:** https://arxiv.org/abs/2602.00053
- **Reference count:** 17
- **Primary result:** On the tested AWS g4dn.xlarge setup, FastAPI had the best single-request latency, while Triton with dynamic batching delivered the highest throughput.

## Executive Summary
This paper compares two Kubernetes-based inference serving patterns for healthcare-oriented NLP inference: a FastAPI CPU service and NVIDIA Triton on a T4 GPU. The benchmark uses a DistilBERT sentiment model on a single AWS g4dn.xlarge instance and evaluates latency and throughput under different serving configurations.

The main takeaway is a deployment tradeoff: FastAPI is strongest as a low-latency, secure edge or gateway layer, while Triton is better suited as the high-volume backend when dynamic batching is enabled. The study frames this as a practical approach for healthcare systems that need both performance and secure handling of PHI.

## Method Summary
The study benchmarks DistilBERT sentiment inference in Kubernetes using two serving approaches on an AWS g4dn.xlarge instance: a FastAPI CPU-based service and NVIDIA Triton on a T4 GPU. Triton is tested both without batching and with dynamic batching, allowing the paper to isolate the impact of batching on runtime behavior.

Reported metrics focus on p50/p95 latency and throughput under the tested workload. The comparison is centered on operational serving performance rather than model accuracy, and the claimed healthcare security advantages are mainly architectural rather than experimentally validated in the benchmark.

## Key Results
- FastAPI delivered the lowest single-request latency: **p50 22 ms**, **p95 45 ms**, **450 req/s**.
- Triton without batching was slower: **p50 28 ms**, **p95 52 ms**, **420 req/s**.
- Triton with dynamic batching reached **780 req/s**, with **p50 34 ms** and **p95 60 ms**; this is **85% over unbatched Triton** and **73% over FastAPI** in throughput.

## Why This Works (Mechanism)
- **FastAPI as a lightweight gateway:** A CPU-based API layer can reduce orchestration overhead and respond quickly for isolated requests, which helps when the main goal is low latency and simpler security control points.
- **Triton + dynamic batching:** Triton can aggregate incoming requests and better utilize GPU parallelism, improving throughput at the cost of slightly higher per-request latency.
- **Healthcare deployment split:** Separating secure request handling from high-throughput inference can support both PHI-sensitive workflows and production-scale serving.

## Foundational Learning
- **Dynamic batching**
  - **Why needed:** Increases GPU efficiency by combining multiple requests into larger execution batches.
  - **Quick check:** Throughput should rise as concurrency increases, while latency may increase modestly.
- **p50 / p95 latency**
  - **Why needed:** Median and tail latency capture both typical and worst-case responsiveness, which matters for clinical workflows.
  - **Quick check:** Compare p50 and p95 together; a small p50 can hide poor tail behavior.
- **Kubernetes serving patterns**
  - **Why needed:** Healthcare deployments often need repeatable scaling, isolation, and rollout control.
  - **Quick check:** Identify which component is exposed externally and which component is responsible for inference execution.
- **CPU vs GPU inference**
  - **Why needed:** Hardware choice strongly affects latency, throughput, and cost profile.
  - **Quick check:** CPU paths often win on simplicity and low overhead; GPU paths often win on batch-heavy throughput.
- **PHI-aware architecture**
  - **Why needed:** Healthcare systems must reduce exposure of sensitive data while preserving operability.
  - **Quick check:** Confirm where authentication, logging, encryption, and data minimization are enforced.

## Architecture Onboarding
- **Component map:** Client -> FastAPI gateway -> Triton inference server -> DistilBERT model -> response
- **Critical path:** Request arrives, is validated/handled by the API layer, routed to the model server, optionally batched by Triton, executed on GPU, then returned to the client.
- **Design tradeoffs:** FastAPI favors simplicity and lower per-request latency; Triton favors higher throughput and better GPU utilization, especially under load. Dynamic batching improves scale but adds queueing delay.
- **Failure signatures:** Rising p95 latency indicates batching/queueing pressure; low throughput suggests poor GPU utilization or insufficient concurrency; a gateway bottleneck points to the API layer rather than the model server.
- **First experiments:**
  1. Run the same DistilBERT workload through FastAPI and Triton at matched concurrency to confirm the latency ordering.
  2. Toggle Triton dynamic batching on/off and measure throughput and tail latency under increasing load.
  3. Insert gateway-style controls in front of Triton and test whether security-oriented layering changes latency materially.

## Open Questions the Paper Calls Out
- The benchmark appears to be a single-model, single-node study (DistilBERT on one AWS g4dn.xlarge instance), so generalizability to other models, payload sizes, traffic patterns, or hospital workloads is limited.
- The reported gains for Triton likely depend on request concurrency and batching configuration; the exact batching settings, load generator details, and warmup protocol are not provided here.
- Security and PHI-handling claims seem architectural rather than empirically validated in the benchmark; it is unclear whether the paper measured threat resistance, auditability, or compliance behavior directly.
- Cost, operational complexity, and GPU utilization efficiency are not fully captured by latency/throughput alone.

## Limitations
- Single-model, single-node benchmark limits generalizability to broader healthcare workloads.
- Security and PHI-readiness are described architecturally, not proven by direct security evaluation.
- Cost, operational complexity, and utilization efficiency are not fully assessed.

## Confidence
- Performance ranking under the tested setup -> **High**
- Dynamic batching improves throughput substantially -> **High**
- FastAPI is the best secure gateway / Triton the better backend for production scale -> **Medium**
- Healthcare security/PHI readiness is demonstrated empirically -> **Low**

## Next Checks
1. Reproduce the benchmark with the same model, instance type, concurrency levels, and batching parameters, then verify p50/p95 latency and throughput.
2. Test additional clinical NLP workloads and different request-size distributions to see whether the FastAPI-vs-Triton ordering holds.
3. Evaluate security controls directly by checking authentication, audit logging, data-in-transit protection, and PHI exposure behavior in the deployed stack.
