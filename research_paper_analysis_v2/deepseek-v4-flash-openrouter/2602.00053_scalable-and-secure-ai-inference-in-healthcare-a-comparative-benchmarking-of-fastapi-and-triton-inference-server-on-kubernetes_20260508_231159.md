# Scalable and Secure AI Inference in Healthcare: A Comparative Benchmarking of FastAPI and Triton Inference Server on Kubernetes

## Quick Facts
- **arXiv ID:** 2602.00053
- **Source URL:** https://arxiv.org/abs/2602.00053
- **Reference count:** 17
- **Primary result:** A hybrid architecture (FastAPI gateway + Triton backend) achieves 780 req/s throughput (73% higher than FastAPI alone) with p50 latency of 34 ms, while maintaining security for PHI de‑identification.

## Executive Summary
This paper benchmarks two deployment strategies for serving a DistilBERT sentiment analysis model in a healthcare setting where both low latency and protected health information (PHI) security are critical. The authors compare a CPU‑based FastAPI server against an NVIDIA Triton Inference Server (GPU) with and without dynamic batching, all running on Kubernetes. They find that while FastAPI offers the lowest single‑request latency (22 ms p50), Triton’s dynamic batching nearly doubles throughput (780 req/s) at only a modest latency increase (34 ms p50). The work validates a hybrid architecture where FastAPI acts as a secure entry point (handling authentication and PHI de‑identification) and Triton serves as the high‑throughput inference backend, arguing this separation of concerns is a best practice for enterprise clinical AI.

## Method Summary
The authors deployed DistilBERT (a 67‑million‑parameter transformer) on two inference stacks: (1) FastAPI running PyTorch on CPU only, and (2) NVIDIA Triton Inference Server running an ONNX‑exported model on a single NVIDIA T4 GPU. Both were containerized and orchestrated via Kubernetes on an AWS g4dn.xlarge instance (4 vCPUs, 16 GB RAM, 1× T4 GPU). Triton’s dynamic batching was configured with a maximum batch size of 16. Load testing used the Locust tool with HTTP/gRPC traffic at concurrency levels of 10, 50, and 100 simulated users. Key metrics were p50 and p95 latency (ms) and throughput (req/s). Horizontal Pod Autoscaling (HPA) was applied to Triton pods (2–10 replicas) based on CPU utilization at 60 %. The hybrid architecture was proposed but not benchmarked end‑to‑end; only individual component performance was measured.

## Key Results
- FastAPI (CPU): p50 latency **22 ms**, throughput **450 req/s**.
- Triton (GPU, no batching): p50 latency **28 ms**, throughput **420 req/s**.
- Triton (GPU, dynamic batch size 16): p50 latency **34 ms**, throughput **780 req/s** (73 % increase over FastAPI).

## Why This Works (Mechanism)

### Mechanism 1 – Hybrid Architecture Separates Security from Inference
- **Claim:** A hybrid architecture (FastAPI gateway + Triton backend) outperforms either system alone for secure, high‑throughput healthcare inference.
- **Mechanism:** FastAPI efficiently handles CPU‑bound, latency‑insensitive security tasks (OAuth2, JWT, PHI de‑identification). Triton maximizes GPU utilization through dynamic batching. The preprocessing microservice strips PHI before requests reach the inference server, preventing sensitive data from being logged.
- **Core assumption:** The workload includes both security‑critical preprocessing and compute‑heavy inference, and the two can be cleanly decoupled via a network boundary.
- **Evidence anchors:**
  - Abstract states “validates the hybrid model as a best practice for enterprise clinical AI.”
  - Section III.B describes FastAPI as the secure entry point; Section III.C describes the decoupled preprocessing.
  - No corpus neighbor directly validates this hybrid claim.
- **Break condition:** If the inference model is lightweight enough to run efficiently on CPU (e.g., a logistic regression), the added complexity of Triton and the hybrid architecture is not warranted.

### Mechanism 2 – Dynamic Batching Increases Throughput
- **Claim:** Triton’s dynamic batching increases throughput by aggregating requests into larger GPU executions with marginal latency increase.
- **Mechanism:** Batching amortises the fixed overhead of GPU kernel launches and data movement. Dynamic batching queues requests for a short window (default ~1 ms) or until the batch is full, then executes as a single batched forward pass. GPU utilisation rises from near‑zero to high (batch size 16).
- **Core assumption:** The model supports dynamic batching (ONNX export with variable batch sizes) and the workload has sufficient concurrent requests to fill batches.
- **Evidence anchors:**
  - Abstract: “Triton achieves superior scalability … delivering a throughput of 780 req/s.”
  - Table I and Section VII.B.2: p50 latency rising from 28 to 34 ms with a 73 % throughput increase.
  - Corpus neighbour “Comparative Analysis of LLM Inference Serving Systems” supports the general batching mechanism.
- **Break condition:** Very low concurrency (single‑user interactive queries) makes batching idle time costly, increasing latency without benefit.

### Mechanism 3 – FastAPI Has Lower Single‑Request Latency
- **Claim:** FastAPI has lower p50 latency than Triton for single‑request workloads due to lower architectural overhead.
- **Mechanism:** FastAPI runs a synchronous CPU inference loop with minimal per‑request overhead (no GPU scheduling, no data transfer, no queue management). For a moderately sized model like DistilBERT, the overhead of Triton’s gRPC/HTTP server, model manager, scheduling, and GPU memory transfer exceeds the computational benefit of GPU acceleration for a single request.
- **Core assumption:** The model is small enough that CPU inference is feasible at low concurrency; GPU communication overhead dominates for single requests.
- **Evidence anchors:**
  - Abstract: “FastAPI provides lower overhead for single‑request workloads with a p50 latency of 22ms.”
  - Section VII.B.1 explains the counter‑intuitive result due to Triton’s server overhead.
  - Corpus neighbour “Green MLOps” uses FastAPI for CPU‑based preprocessing, consistent with this mechanism.
- **Break condition:** For very large models (GPT‑style LLMs) that cannot run on CPU, this comparison does not apply; Triton is the only viable option.

## Architecture Onboarding

### Component Map (plain text)
Gateway Layer (FastAPI pod: OAuth2, JWT, request routing) → Preprocessing Layer (independent microservice pod: PHI de‑identification, data normalization) → Inference Layer (Triton pod: ONNX model, dynamic batching, gRPC/HTTP endpoints) → Orchestration (Kubernetes with HPA 2‑10 replicas, CPU threshold 60 %) → Load generation (Locust: HTTP/gRPC traffic, concurrency 10/50/100).

### Critical Path
External request → FastAPI gateway (authentication, routing) → Preprocessor microservice (PHI stripping, normalization) → Triton Inference Server (dynamic batching, GPU execution) → Response back through gateway.

### Design Tradeoffs
- **Latency vs. throughput:** Triton dynamic batching increases p50 latency from 22 ms to 34 ms but increases throughput 73 %.
- **Security vs. complexity:** Hybrid architecture adds a preprocessing microservice that must be managed and scaled independently.
- **Ease of deployment vs. performance:** FastAPI‑only is simpler but cannot scale to 780 req/s.

### Failure Signatures
- **p95 latency spikes > 60 ms:** Likely dynamic batching timeout too large or batch size too high.
- **All requests failing:** Preprocessor pod crash (PHI de‑identification failure) or authentication token expired.
- **Throughput drop:** HPA scale‑in due to CPU threshold false positive on GPU workload.

### First 3 Experiments
1. **Reproduce Table I:** Deploy DistilBERT on FastAPI (CPU) and Triton (GPU, ONNX). Use Locust with 50 concurrent users, measure p50/p95 latency and throughput for FastAPI, Triton no batching, and Triton batch size 16.
2. **Test gateway failure:** Disable preprocessor microservice and observe whether PHI remains in inference server logs. This validates the security decoupling claim.
3. **Measure batching overhead:** Run Triton with batch size 1, 2, 4, 8, 16 at 100 concurrent users. Plot p50 latency vs. throughput to find the optimal batch size for a given latency budget.

## Open Questions the Paper Calls Out

### Open Question 1
**How does the hybrid architecture’s performance (latency and throughput) compare to using FastAPI or Triton alone when measured end‑to‑end, including preprocessing and PHI de‑identification overhead?**  
- *Basis in paper:* The authors propose a hybrid architecture but benchmark each component in isolation, not the combined pipeline.  
- *Why unresolved:* The preprocessing microservice and secure gateway introduce additional latency and potential bottlenecks not captured in the individual benchmarks.  
- *What evidence would resolve it:* A controlled experiment measuring p50/p95 latency and throughput for the full hybrid pipeline under identical load conditions.

### Open Question 2
**Under what specific utilisation thresholds and request patterns does GPU‑aware autoscaling (using NVIDIA DCGM metrics) outperform CPU‑based HPA for Triton deployments in healthcare settings?**  
- *Basis in paper:* The authors note their HPA configuration relied on CPU utilisation and call for future work on GPU‑aware autoscaling using NVIDIA DCGM metrics.  
- *Why unresolved:* The paper does not test or simulate GPU‑aware scaling; optimal thresholds and trade‑offs remain unknown.  
- *What evidence would resolve it:* A benchmark comparing response time, resource waste, and cost under realistic burst patterns (e.g., clinic hours vs. nightly EHR batch processing) for both scaling strategies.

### Open Question 3
**How does the hybrid architecture’s latency and compliance performance degrade when deployed on edge devices with limited GPU resources (e.g., in‑hospital servers with a single low‑power GPU)?**  
- *Basis in paper:* The paper lists adapting the stack for lightweight edge devices as a future challenge, noting the current design is for cloud/data center clusters.  
- *Why unresolved:* The hardware used (NVIDIA T4 GPU on AWS) is relatively powerful; edge devices may have lower GPU memory, slower interconnects, and different batching behaviour.  
- *What evidence would resolve it:* Repeat the benchmarking experiment on representative edge hardware (e.g., NVIDIA Jetson series) with the same DistilBERT model and load scenarios.

## Limitations
- **Reproducibility gaps (Medium):** Exact DistilBERT ONNX export parameters (dynamic axes, opset) and Locust load script details (think time, payload) are unspecified, making exact latency/throughput replication uncertain.
- **Single‑model scope (High):** Only DistilBERT is benchmarked. Results may not generalise to larger models (e.g., GPT‑scale LLMs) where GPU is mandatory or smaller models where CPU overhead dominates.
- **Ambiguous security validation (Low‑Medium):** The claim that PHI is never logged by Triton is asserted but not empirically validated (e.g., no audit of inference server logs after preprocessing failure).
