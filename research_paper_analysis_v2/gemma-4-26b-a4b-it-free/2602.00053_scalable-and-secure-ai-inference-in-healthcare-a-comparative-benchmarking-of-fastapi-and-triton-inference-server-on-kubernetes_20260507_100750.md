# Scalable and Secure AI Inference in Healthcare: A Comparative Benchmarking of FastAPI and Triton Inference Server on Kubernetes

## Quick Facts
* **arXiv ID:** 2602.00053
* **Source URL:** https://arxiv.org/abs/2602.00053
* **Reference Count:** 17
* **Primary Result:** FastAPI excels at low-latency single requests, while Triton with dynamic batching provides significantly higher throughput for high-volume production workloads.

## Executive Summary
This paper investigates the optimal deployment strategy for healthcare machine learning inference, balancing the competing requirements of low latency, high scalability, and secure Protected Health Information (PHI) handling. The study compares two distinct Kubernetes-based serving architectures to determine how to best manage computational resources in a regulated environment.

By benchmarking a DistilBERT model on AWS g4dn.xlarge instances, the research identifies a hybrid architectural pattern. The findings suggest that a decoupled approach—using FastAPI as a secure, low-latency gateway and NVIDIA Triton as a high-throughput backend—offers the most robust solution for production-grade healthcare AI services.

## Method Summary
The researchers conducted a comparative performance benchmark using a DistilBERT sentiment analysis model deployed on Kubernetes. The testing environment utilized AWS g4dn.xlarge instances, allowing for a direct comparison between CPU-based and GPU-based execution.

Two primary serving configurations were evaluated: a FastAPI service running on the CPU and an NVIDIA Triton Inference Server running on a T4 GPU. The Triton configuration was further subdivided into two testing scenarios: standard inference without batching and inference utilizing Triton's dynamic batching capabilities. Performance was measured across latency (p50 and p95) and throughput (requests per second).

## Key Results
* **Low Latency Winner:** FastAPI achieved the lowest single-request latency (p50: 22 ms, p95: 45 ms) and a throughput of 450 req/s.
* **Throughput Winner:** Triton with dynamic batching reached 780 req/s, representing a 73% throughput advantage over FastAPI and an 85% gain over unbatched Triton.
* **Optimal Hybrid Model:** The study concludes that FastAPI is best suited as a secure gateway, while Triton serves as the superior backend for high-volume inference.

## Why This Works (Mechanism)
The performance divergence is driven by how each server manages request processing. FastAPI, being a lightweight ASGI framework, minimizes the overhead for individual, asynchronous requests, making it ideal for immediate, low-latency responses. In contrast, Triton leverages GPU parallelism; by using dynamic batching, it aggregates multiple incoming requests into a single execution cycle, maximizing the utilization of the T4 GPU's tensor cores and significantly increasing total throughput at the cost of slightly higher individual request latency.

## Foundational Learning
* **Dynamic Batching:** Needed to maximize GPU utilization by grouping requests. *Quick check: Does throughput increase as p50 latency rises?*
* **Inference Latency (p50/p95):** Needed to understand the user experience and tail-end delays. *Quick check: Is the gap between p50 and p95 widening under load?*
* **Throughput (req/s):** Needed to determine the system's capacity for scaling. *Quick check: Does throughput scale linearly with batch size?*
* **ASGI/FastAPI Overhead:** Needed to evaluate the efficiency of the web gateway. *Quick check: How much CPU is consumed by request parsing vs. model logic?*
* **Kubernetes Orchestration:** Needed to manage containerized scaling and resource allocation. *Quick check: Are pods scaling based on CPU/GPU utilization or request count?*

## Architecture Onboarding
**Component Map**
Client Request -> FastAPI Gateway (Security/PHI Handling) -> Triton Inference Server (GPU Backend) -> DistilBERT Model

**Critical Path**
Request Authentication -> PHI Sanitization/Encryption -> Network Transfer (Gateway to Backend) -> Batching Queue -> GPU Execution -> Result Decryption/Return.

**Design Tradeoffs**
* **Latency vs. Throughput:** Choosing FastAPI optimizes for the fastest individual response; choosing Triton with batching optimizes for the most requests processed per second.
* **Complexity vs. Security:** A decoupled gateway/backend architecture increases operational complexity but provides a dedicated layer for security and PHI compliance.

**Failure Signatures**
* **Gateway Bottleneck:** High CPU usage on FastAPI pods with low GPU utilization on Triton.
* **Batching Lag:** High p95 latency in Triton despite high throughput, indicating excessively large batch windows.
* **Resource Starvation:** Increasing p50 latency across both tiers indicating Kubernetes node saturation.

**First Experiments**
1. Measure the latency delta when adding mTLS encryption to the FastAPI gateway.
2. Vary the dynamic batching timeout in Triton to find the "sweet spot" between throughput and p95 latency.
3. Stress test the FastAPI gateway to determine the maximum request volume it can handle before becoming a bottleneck for the Triton backend.

## Limitations
* **Narrow Model Scope:** Results are specific to DistilBERT and may not generalize to LLMs where KV caching and memory management are primary bottlenecks.
* **Hardware Specificity:** Performance trade-offs are tied to the AWS g4dn.xlarge instance and may shift on different compute ratios.
* **Security Measurement:** The study benchmarks performance rather than empirically testing the specific latency overhead of security protocols (e.g., encryption).

## Confidence
* **Performance Benchmarking (FastAPI vs. Triton):** High
* **Architectural Recommendation (Gateway vs. Backend):** Medium
* **Generalizability to Healthcare Workloads:** Low

## Next Checks
1. **Cross-Model Scaling:** Replicate the benchmark using a larger model (e.g., Llama-3-8B) to determine if Triton's throughput advantage increases or if memory bottlenecks shift the optimal batch size.
2. **Security Overhead Audit:** Quantify the specific latency penalty introduced by implementing mTLS or PHI encryption layers within the FastAPI gateway.
3. **Resource Heterogeneity Test:** Test the deployment on different instance classes (e.g., CPU-optimized vs. GPU-optimized) to see if the FastAPI advantage persists as compute density increases.
