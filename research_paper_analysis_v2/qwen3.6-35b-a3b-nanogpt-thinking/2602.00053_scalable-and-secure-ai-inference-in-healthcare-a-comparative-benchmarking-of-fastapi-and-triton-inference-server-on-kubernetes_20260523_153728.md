# Scalable and Secure AI Inference in Healthcare: A Comparative Benchmarking of FastAPI and Triton Inference Server on Kubernetes

## Quick Facts
- arXiv ID: 2602.00053
- Source URL: https://arxiv.org/abs/2602.00053
- Reference Count: 17
- Primary Result: A hybrid FastAPI-Triton architecture achieves a 73% throughput increase (780 req/s) with only a 12 ms p50 latency overhead while maintaining PHI de-identification boundaries.

## Executive Summary
This study addresses the critical need for scalable, low-latency, and HIPAA-compliant AI inference in healthcare by benchmarking a lightweight FastAPI REST service against NVIDIA Triton Inference Server on Kubernetes. Using a DistilBERT sentiment analysis model under controlled concurrent loads, the research demonstrates a clear performance trade-off between specialized inference servers and general-purpose web frameworks. Triton leverages dynamic batching to significantly aggregate throughput, while the standard FastAPI service exhibits lower per-request latency but lacks hardware acceleration efficiency.

Crucially, the authors validate a hybrid architectural pattern that isolates PHI de-identification and authentication within a FastAPI security gateway, routing normalized tensors to a Triton GPU backend. This design preserves regulatory compliance without degrading downstream inference performance, establishing the hybrid configuration as an optimal standard for regulated, high-throughput clinical AI deployments where both security and scalability are mandatory.

## Method Summary
Researchers deployed containerized inference services on an AWS g4dn.xlarge instance equipped with a single NVIDIA T4 GPU, utilizing a lightweight Kubernetes runtime for orchestration. The evaluation framework containerized a single-worker FastAPI service for CPU-bound inference and security routing, alongside a Triton Inference Server configured to serve a preconverted DistilBERT model (ONNX) with dynamic batching enabled. Controlled load generation was executed using Locust scripts, simulating concurrent user loads at 10, 50, and 100 virtual users to establish baseline performance thresholds.

The testing methodology isolated single-request sequential execution from batched execution to quantify architectural overhead. Metrics captured included median (p50) and tail (p95) latency, alongside aggregate throughput (requests per second). By comparing FastAPI and Triton across varying concurrency levels and batch configurations, the study systematically mapped the latency-throughput trade-offs inherent to GPU memory allocation, kernel launch overhead, and request scheduling mechanisms.

## Key Results
- Triton Inference Server achieved 780 requests per second with a 34 ms p50 latency, representing a 73% throughput increase over the FastAPI baseline.
- FastAPI demonstrated a lower p50 latency of 22 ms and handled 450 req/s but lacked the dynamic batching and GPU acceleration necessary for sustained high concurrency.
- The validated hybrid architecture successfully decoupled PHI de-identification in a FastAPI gateway from GPU computation in Triton, maintaining compliance boundaries without impacting inference throughput.

## Why This Works (Mechanism)

### Mechanism 1: Dynamic Batching for Throughput Scaling
- Claim: Triton achieves higher aggregate throughput than general-purpose web frameworks under concurrent clinical loads.
- Mechanism: Dynamic batching aggregates individual HTTP/gRPC requests into larger tensor operations, amortizing fixed GPU kernel launch and memory allocation overhead across multiple samples.
- Core assumption: Clinical inference workloads can tolerate controlled millisecond-level queuing delays to fill batch windows without violating real-time decision support SLAs.
- Evidence anchors: The study reports Triton reaching 780 req/s with only a marginal latency increase when shifting to batch size 16. Performance scales dramatically with volume, confirming that queue filling effectively masks per-sample overhead.
- Break condition: Workload concurrency drops below the threshold required to fill batch windows, causing queue starvation and throughput collapse.

### Mechanism 2: Separation of Concerns for PHI Compliance
- Claim: Isolating PHI de-identification behind a gateway reduces regulatory exposure without degrading downstream inference throughput.
- Mechanism: The FastAPI gateway handles authentication and routes plain-text inputs to a dedicated CPU-bound preprocessing microservice that strips or obfuscates protected data before forwarding normalized tensors to the GPU backend.
- Core assumption: CPU-bound NLP normalization pipelines can absorb peak ingress traffic surges without creating a systemic bottleneck relative to GPU compute limits.
- Evidence anchors: Validation confirms that stripping sensitive data in a dedicated preprocessing layer enforces architectural boundaries, allowing independent auditing of the compliance plane separate from compute optimization.
- Break condition: Preprocessing microservice pod failures or memory exhaustion under burst traffic, causing cascading retries to the gateway and degraded p95 latency.

### Mechanism 3: Architectural Overhead in Sequential Inference
- Claim: Specialized inference servers may exhibit higher p50 latency than lightweight web frameworks for sequential or low-concurrency traffic.
- Mechanism: Triton's request scheduling, protocol handling, and GPU VRAM context switching introduce fixed architectural overhead that temporarily outweighs acceleration benefits for single-sample execution.
- Core assumption: The latency cost of host-to-device data movement remains relatively constant per request regardless of model depth or complexity.
- Evidence anchors: FastAPI achieved 22 ms p50 latency on single requests, outperforming Triton's 34 ms baseline where GPU scheduling overhead dominated execution time.
- Break condition: Throughput demands exceed the GPU's compute queue capacity, causing p95/tail latency to spike exponentially beyond acceptable clinical thresholds.

## Architecture Onboarding
Component Map:
FastAPI Gateway (Auth/Routing) -> NLP/CV Preprocessor Pod (PHI de-id/Normalization/Tensor formatting) -> Triton Inference Server (ONNX/TorchScript on NVIDIA T4 GPU) -> Kubernetes Orchestration (HPA, Liveness/Readiness probes, TLS)

Critical Path:
Ingress traffic -> JWT validation in FastAPI -> PHI stripping in Preprocessor -> gRPC/HTTP request to Triton -> GPU kernel execution -> tensor serialization -> response return to gateway -> client

Design Tradeoffs:
Accepts higher p95 tail latency (queue waiting for batch windows) to gain 73% higher throughput; sacrifices single-request latency optimization (FastAPI's 22ms) for enterprise scalability; decouples compliance logic from ML compute to allow independent scaling and security auditing.

Failure Signatures:
p95 latency plateauing above 60ms without throughput growth indicates batch window misconfiguration; Gateway 503/timeout storms signal preprocessor memory saturation; Triton readiness probe failures correlate with GPU ECC errors or OOM kernel crashes.

First 3 Experiments:
1. Tune dynamic batching patience and max batch size to identify the inflection point where p95 latency exceeds clinical SLA (target: 50ms) before throughput gains flatten.
2. Inject synthetic PHI-rich text payloads to measure preprocessing microservice throughput vs. Triton's GPU compute limit, confirming the gateway/preprocessor does not become the system bottleneck.
3. Simulate HPA-triggered pod scaling during a sustained 800 req/s load to validate zero-downtime rolling updates and measure stateless request redistribution latency.

## Open Questions the Paper Calls Out
Open Question 1: How does GPU-aware autoscaling using NVIDIA DCGM metrics improve latency reliability compared to CPU-based HPA in healthcare Kubernetes deployments?
- Basis in paper: Section IX explicitly identifies CPU-based scaling as a lagging indicator for GPU workloads and calls for GPU-aware alternatives.
- Why unresolved: The study only tested CPU-based HPA and did not benchmark GPU duty cycles or memory utilization as scaling triggers.
- What evidence would resolve it: Controlled load tests comparing scaling latency, request queuing, and cost efficiency under variable traffic using DCGM-driven versus CPU-driven autoscaling.

Open Question 2: What architectural modifications enable this hybrid FastAPI-Triton pattern to function on resource-constrained in-hospital edge servers without compromising HIPAA compliance?
- Basis in paper: Section IX notes adapting the cloud-designed stack to lightweight edge devices remains a stated challenge.
- Why unresolved: Experiments ran on a standard cloud instance, leaving offline operation, memory constraints, and edge network latency untested.
- What evidence would resolve it: Benchmarking on edge hardware (e.g., NVIDIA Jetson) measuring throughput degradation, PHI audit integrity, and failover performance under unstable connectivity.

Open Question 3: How can automated statistical monitoring be integrated into this pipeline to detect clinical data distribution shifts and mitigate model drift risks?
- Basis in paper: Section IX states the pipeline lacks automated drift detection and identifies it as critical for patient safety.
- Why unresolved: The benchmark used a static model without simulating data shifts or testing monitoring integration.
- What evidence would resolve it: Deployment of a drift-detection sidecar during sustained inference, measuring detection accuracy, false positive rates, and the impact on clinical prediction performance under evolving data.

## Limitations
- Benchmarks operate on a single T4 GPU with a lightweight NLP model, leaving generalization to larger LLMs, vision transformers, or multi-node clusters unverified.
- HIPAA alignment is inferred from architectural data-plane isolation rather than independent security audits, penetration testing, or validated audit trails.
- The preliminary posting status and near-zero citation footprint classify findings as early-stage; exact serving configurations and load-test harnesses remain partially unspecified.

## Confidence
- Triton dynamic batching throughput gains: High
- Hybrid PHI-gateway isolation efficacy: Medium
- Fixed clinical SLA applicability (static latency thresholds): Low
