# Scalable and Secure AI Inference in Healthcare: A Comparative Benchmarking of FastAPI and Triton Inference Server on Kubernetes
## Quick Facts
- arXiv ID: 2602.00053
- Source URL: https://arxiv.org/abs/2602.00053
- Reference count: 17
- Primary result: A hybrid FastAPI-Triton architecture on Kubernetes delivers a 73% throughput increase over FastAPI alone while maintaining HIPAA-compliant PHI de-identification and clinical-grade latency.

## Executive Summary
This paper benchmarks scalable and HIPAA-compliant AI inference for clinical NLP workloads on Kubernetes, comparing a lightweight FastAPI REST service against NVIDIA Triton Inference Server. The study evaluates a DistilBERT sentiment analysis model under controlled concurrent loads to quantify performance trade-offs between CPU-based serving and GPU-accelerated dynamic batching. 

The authors validate a hybrid deployment pattern that decouples security preprocessing from model execution, addressing strict regulatory requirements without sacrificing throughput. By routing sensitive clinical text through a dedicated de-identification gateway before model inference, the architecture mitigates data leakage risks while leveraging Triton's tensor aggregation to maximize hardware utilization.

## Method Summary
The evaluation focuses on inference-only benchmarking using a pretrained DistilBERT model exported to PyTorch for FastAPI and ONNX for Triton. Experiments are conducted on a cloud Kubernetes cluster provisioned with a single GPU node. Load testing simulates 10, 50, and 100 concurrent users via Locust scripts and a custom Python client, routing traffic through both standalone FastAPI endpoints and Triton with batch sizes of 1 and 16. 

System scaling is managed by Kubernetes Horizontal Pod Autoscaling targeting 60% CPU utilization, while latency percentiles and request throughput are captured using Prometheus metrics and custom telemetry. The pipeline enforces input validation and PHI stripping in the gateway layer, ensuring compliance before data reaches the inference engine.

## Key Results
- FastAPI baseline achieves 450 req/s with 22 ms p50 latency but lacks hardware acceleration and dynamic batching capabilities.
- Triton with dynamic batching (size 16) reaches 780 req/s with 34 ms p50 latency, yielding a 73% throughput increase with acceptable latency overhead.
- A hybrid FastAPI gateway for PHI de-identification combined with a Triton backend successfully enforces HIPAA compliance while preserving GPU throughput gains.

## Why This Works (Mechanism)
### Mechanism 1
- Claim: Triton inference throughput scales with concurrent load via dynamic request aggregation, yielding ~73% higher throughput than FastAPI under peak conditions.
- Mechanism: Dynamic batching aggregates independent requests into tensor groups (e.g., batch size 16) before GPU execution, amortizing kernel launch overhead and maximizing GPU compute utilization.
- Core assumption: The latency penalty from queuing requests to fill a batch does not exceed the clinical acceptability threshold for the target workload.
- Evidence anchors: Abstract and Section VII.B.2 confirm the 73% throughput increase and 780 req/s peak at batch size 16 with marginal latency overhead.

### Mechanism 2
- Claim: A hybrid architecture combining FastAPI as a gateway and Triton as a backend enables HIPAA-compliant inference without sacrificing the throughput gains of GPU acceleration.
- Mechanism: Decoupling security logic (PHI de-identification, auth) into a FastAPI gateway and preprocessing microservice ensures sensitive data is stripped or obfuscated before reaching the inference layer, reducing leakage risk in model logs while preserving Triton's compute isolation.
- Core assumption: De-identification and routing logic can execute within latency bounds that prevent the gateway from becoming the bottleneck when Triton is saturated.
- Evidence anchors: Abstract and Section VIII.B validate the separation of concerns and risk mitigation through pre-inference data stripping.

### Mechanism 3
- Claim: Kubernetes HPA provides elastic scaling for the inference layer, though current CPU-based metrics may lag under GPU-bound saturation.
- Mechanism: Horizontal Pod Autoscaling monitors resource utilization and adjusts replica counts; however, CPU metrics act as a lagging indicator for GPU workloads, potentially causing delayed scaling responses during rapid traffic bursts.
- Core assumption: Future iterations using GPU-aware metrics (e.g., DCGM) are required to resolve the latency gap between resource saturation and pod provisioning.
- Evidence anchors: Sections V and IX.1 note the reliance on CPU thresholds and explicitly flag GPU-aware autoscaling as a necessary future improvement.

## Architecture Onboarding
- Component map: FastAPI Gateway -> Preprocessing Microservice -> Triton Inference Server -> Kubernetes Orchestration
- Critical path: Client Request -> FastAPI (Auth/Routing) -> Preprocessor (De-id/Norm) -> Triton (Batch/Inference) -> Response
- Design tradeoffs:
  - Batching throughput vs. increased tail latency (p95)
  - FastAPI flexibility/security vs. Triton raw performance
  - CPU-based HPA simplicity vs. GPU scaling responsiveness
- Failure signatures:
  - p95 latency spikes indicating batch queue saturation
  - HPA lag causing throughput drops during GPU bursts
  - PHI leakage if preprocessor fails or bypasses de-identification

## Open Questions the Paper Calls Out
### Open Question 1
- Question: How does implementing GPU-aware autoscaling via NVIDIA DCGM metrics affect scaling latency and resource efficiency for Triton inference workloads compared to standard CPU-based HPA?
- Basis in paper: Section IX identifies that current HPA relies on CPU utilization, a lagging indicator for GPU workloads, and explicitly calls for GPU-aware autoscaling as future work.
- Why unresolved: Kubernetes default autoscaling triggers on CPU thresholds, failing to capture rapid GPU utilization spikes common in batched deep learning inference.
- What evidence would resolve it: Comparative benchmarking of scaling response time, tail latency, and GPU duty cycle efficiency under controlled load bursts using DCGM-based vs. CPU-based HPA policies.

### Open Question 2
- Question: What are the performance and security trade-offs of deploying the hybrid FastAPI-Triton architecture on resource-constrained hospital edge devices?
- Basis in paper: Section IX notes that adapting the cloud-validated stack for lightweight edge devices remains a challenge and requires future investigation.
- Why unresolved: The current validation assumes data center infrastructure with abundant RAM and stable networking, which differs significantly from clinical edge constraints.
- What evidence would resolve it: Empirical deployment on representative in-hospital edge hardware, measuring PHI de-identification overhead, inference latency, throughput, and fault tolerance under intermittent connectivity.

### Open Question 3
- Question: How can automated statistical drift detection be integrated into the inference pipeline to monitor clinical data distribution shifts without degrading real-time latency?
- Basis in paper: Sections III and IX state that the current pipeline lacks automated drift detection and emphasize its necessity for patient safety and model reliability.
- Why unresolved: Statistical monitoring introduces computational overhead and architectural complexity that has not been benchmarked against the strict low-latency requirements of clinical decision support.
- What evidence would resolve it: Implementation of lightweight drift algorithms in the preprocessing layer, with longitudinal evaluation of detection accuracy, false positive rates, and impact on p50/p95 latency.

## Limitations
- Benchmark scope is restricted to a single DistilBERT sentiment workload and a fixed cloud node, limiting generalizability to larger clinical LLMs or multi-node clusters.
- Critical pipeline components like the PHI de-identification engine and Locust load-testing configuration remain opaque, complicating independent validation.
- Kubernetes HPA relies on lagging CPU metrics, introducing scaling delays during GPU-bound traffic bursts that are not addressed in the current design.

## Confidence
- Dynamic batching throughput gains: High
- Hybrid PHI-gateway compliance architecture: Medium (opaque de-id implementation)
- GPU-bound autoscaling responsiveness: Low (explicitly flagged as a known limitation)
