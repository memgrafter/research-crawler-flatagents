# Scalable and Secure AI Inference in Healthcare: A Comparative Benchmarking of FastAPI and Triton Inference Server on Kubernetes

## Quick Facts

- **arXiv ID:** 2602.00053
- **Source URL:** https://arxiv.org/abs/2602.00053
- **Reference count:** 17
- **Primary result:** FastAPI had the lowest single-request latency, while Triton with dynamic batching delivered the highest throughput; the paper recommends a hybrid FastAPI gateway plus Triton backend design.

## Executive Summary

This paper benchmarks two Kubernetes-based approaches for healthcare AI inference: a Python FastAPI REST service running DistilBERT on CPU, and NVIDIA Triton Inference Server running the same model on GPU. The central deployment question is how to balance low latency, high throughput, scalability, and healthcare security controls such as PHI handling.

The reported results favor FastAPI for low-concurrency, single-request latency, but favor Triton once dynamic batching can keep the GPU busy. The paper’s recommended architecture is hybrid: FastAPI handles authentication, routing, validation, and PHI de-identification, while Triton performs high-throughput backend inference.

## Method Summary

The benchmark uses a pretrained DistilBERT sentiment analysis model served through two containerized paths on Kubernetes. The FastAPI path loads the model in PyTorch and performs CPU inference through a `/predict` endpoint. The Triton path serves an exported model, likely ONNX, on an NVIDIA T4 GPU, with configurations tested both without batching and with dynamic batching at batch size 16.

Load testing is performed with Locust at varying concurrency levels, including 10, 50, and 100 concurrent users. The main metrics are p50 latency, p95 latency, and throughput in requests per second. The paper does not describe model training; the focus is inference deployment and serving behavior.

## Key Results

- **FastAPI CPU, batch 1:** p50 22 ms, p95 45 ms, 450 req/s — best single-request latency.
- **Triton GPU, no batching:** p50 28 ms, p95 52 ms, 420 req/s — added serving/GPU overhead for individual requests.
- **Triton GPU, dynamic batch size 16:** p50 34 ms, p95 60 ms, 780 req/s — about 73% higher throughput than FastAPI.

## Why This Works (Mechanism)

### Mechanism 1: FastAPI can win for simple single-request inference

- **Claim:** FastAPI can be preferable for low-concurrency, single-request inference when the workload is CPU-tolerant and request-handling overhead dominates compute time.
- **Mechanism:** The serving path is relatively direct: HTTP request → Python validation/routing → CPU PyTorch inference → response. In this setup, avoiding Triton scheduling and GPU transfer overhead appears to reduce median and tail latency for batch size 1.
- **Core assumption:** DistilBERT CPU inference is small enough that GPU acceleration does not offset Triton/GPU orchestration overhead for individual requests.
- **Evidence anchors:** FastAPI reports p50 22 ms versus Triton p50 28 ms without batching. The paper attributes Triton’s single-request overhead to inference scheduling and GPU data movement.
- **Break condition:** This may stop holding for larger models, CPU saturation, optimized GPU transfer, dominant preprocessing costs, or a FastAPI path that adds efficient GPU batching.

### Mechanism 2: Triton improves throughput when dynamic batching is effective

- **Claim:** Triton becomes more effective when request volume is high enough to form useful batches and keep the GPU busy.
- **Mechanism:** Dynamic batching groups incoming requests into one GPU execution. This amortizes scheduling and transfer overhead across multiple examples, increasing GPU utilization and throughput while adding modest queueing delay.
- **Core assumption:** Traffic is sufficiently steady, batchable, and concurrent, and the GPU has unused parallel capacity.
- **Evidence anchors:** Triton with dynamic batching reaches 780 req/s on a single NVIDIA T4 GPU, compared with 450 req/s for FastAPI and 420 req/s for Triton without batching.
- **Break condition:** Benefits weaken with sparse traffic, strict latency SLOs, poorly tuned queue delays, variable input shapes, GPU memory limits, or bottlenecks in preprocessing/gateway stages.

### Mechanism 3: A hybrid architecture separates security control from inference acceleration

- **Claim:** A FastAPI gateway plus Triton backend can separate healthcare security and routing responsibilities from compute-heavy model execution.
- **Mechanism:** FastAPI handles authentication, routing, validation, timeouts, and PHI de-identification orchestration before requests reach Triton. Triton then focuses on optimized model execution.
- **Core assumption:** PHI de-identification is correctly implemented and enforced before Triton receives payloads.
- **Evidence anchors:** The paper proposes FastAPI as a secure gateway for PHI de-identification and Triton as the inference backend. It describes OAuth2/JWT authentication, routing, retries/timeouts, preprocessing services, and model-serving separation.
- **Break condition:** The design fails if PHI stripping is incomplete, logs capture raw inputs, service-to-service security is weak, Kubernetes secrets are mishandled, or Triton is exposed directly.

## Foundational Learning

### Concept: Latency, tail latency, and throughput

- **Why needed:** The recommendation depends on distinguishing p50 latency, p95 latency, and requests per second. FastAPI wins on single-request latency, while Triton wins on batched throughput.
- **Quick check:** If p50 improves but p95 worsens, what kind of user or clinical workflow is most affected?

### Concept: Dynamic batching and GPU utilization

- **Why needed:** Triton’s main advantage comes from batching requests so the GPU processes more examples per execution.
- **Quick check:** Why can batching increase throughput while also increasing p95 latency?

### Concept: CPU versus GPU serving overhead

- **Why needed:** GPU acceleration is not automatically faster for every request. Small models or short inputs may be dominated by transfer, scheduling, and framework overhead.
- **Quick check:** When might a CPU service beat a GPU service for inference latency?

- **Why needed:** The benchmark depends on containerized services, probes, scaling behavior, resource requests, routing, and service discovery.

## Architecture Onboarding

### Component map

External client -> FastAPI gateway -> OAuth2/JWT authentication and authorization -> request validation and routing -> NLP/CV preprocessing microservice -> PHI de-identification and normalization -> Triton Inference Server over HTTP/gRPC -> versioned ONNX/TorchScript model repository -> metrics endpoint -> Kubernetes HPA, probes, and service routing

### Critical path

For an NLP request, the client sends text to FastAPI. FastAPI validates the token, records only safe metadata, routes the request to preprocessing, removes or obfuscates PHI, tokenizes and normalizes the input into model tensors, sends the tensors to Triton, receives the prediction, maps model output into an API response, and returns it to the client. Latency should be measured at each hop.

### Design tradeoffs

- **FastAPI:** Flexible, simple Python integration, good for authentication and custom request handling, and lower single-request overhead in this benchmark. It lacks specialized GPU-serving features.
- **Triton:** More complex, but provides optimized model serving, GPU execution, dynamic batching, and higher throughput under concurrency.
- **PHI preprocessing:** Improves control over what reaches inference services, but adds a service hop and another failure point.
- **Autoscaling:** CPU-based HPA is easy to configure, but may be a poor proxy for GPU saturation and batching efficiency.

### Failure signatures

- Rising p95 with stable p50 may indicate batching queue delays or downstream congestion.
- Triton throughput below expectation may indicate ineffective batching, GPU underutilization, wrong model format, or small batch windows.
- FastAPI 401/403 spikes suggest authentication or token issues.
- Increased 5xx after deployment may indicate readiness probe, model-load, or service-discovery problems.
- PHI appearing in logs is a critical compliance failure requiring immediate rollback and log containment.

### First 3 experiments

1. Reproduce the reported baseline: run Locust at 10, 50, and 100 concurrent users for FastAPI CPU, Triton batch size 1, and Triton dynamic batch size 16; capture p50, p95, req/s, CPU, GPU utilization, and error rate.
2. Tune Triton batching: vary preferred batch sizes and max queue delay; plot throughput versus p95 to identify the highest throughput that still satisfies the clinical latency SLO.
3. Validate the security path: send synthetic PHI-containing payloads, confirm de-identification before Triton, inspect gateway/preprocessor/Triton logs, verify auth failures are blocked, and ensure Triton is not directly reachable outside the cluster.

## Open Questions the Paper Calls Out

### Open Question 1: GPU-aware autoscaling

- **Question:** Does GPU-aware autoscaling using NVIDIA DCGM metrics reduce latency and improve utilization compared with CPU-based HPA for Triton workloads?
- **Basis in paper:** Future work calls CPU utilization a lagging indicator and proposes GPU-aware autoscaling.
- **Why unresolved:** The benchmark used CPU-based HPA only.
- **Evidence needed:** Controlled Kubernetes tests comparing CPU-HPA versus DCGM-based HPA under bursty healthcare inference loads.

### Open Question 2: Edge deployment

- **Question:** Can the FastAPI–Triton healthcare inference stack meet latency, throughput, and security requirements on hospital edge hardware?
- **Basis in paper:** The authors identify edge deployment adaptation as a remaining challenge.
- **Why unresolved:** Experiments were limited to an AWS g4dn.xlarge cloud instance.
- **Evidence needed:** Benchmarks on representative in-hospital servers or edge GPUs, including failure recovery and PHI controls.

### Open Question 3: Model-drift detection

- **Question:** How can automated model-drift detection be integrated into the pipeline to maintain clinical safety?
- **Basis in paper:** The pipeline lacks automated drift detection, and statistical monitoring is named as a critical next step.
- **Why unresolved:** No drift metrics, thresholds, or monitoring architecture were evaluated.
- **Evidence needed:** Prospective tests detecting distribution shifts in clinical text and triggering validated alerts or retraining workflows.

## Limitations

- Exact model checkpoint, tokenizer settings, input length distribution, Triton configuration, pod resources, replica counts, warmup duration, and Locust behavior are not fully specified.
- The evidence is mostly internal to the reported benchmark; related corpus support is adjacent and does not independently validate these exact measurements.
- Security claims are architectural rather than compliance-proven; PHI safety depends on implementation quality, logging controls, access control, network isolation, and auditability.
